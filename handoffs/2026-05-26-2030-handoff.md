## Blocages / échecs

### Problème partiellement résolu : "Failed to fetch" — fausse alerte

**Découverte clé (20:16) :** L'utilisateur a reçu un email de devis à 20h16 malgré l'alerte "Failed to fetch". Le serveur traite bien la requête jusqu'au bout (données sauvées + PDF généré + emails envoyés), mais la réponse HTTP n'arrive pas au navigateur.

**Diagnostic révisé :** Ce n'est PAS un crash serveur. C'est une **socket fermée pendant la génération PDF**. Puppeteer prend 5-10 secondes. Durant ce temps, la connexion HTTP est coupée côté navigateur (ou réseau local). Quand `res.json({ success: true })` est appelé, la socket est morte → le navigateur voit "Failed to fetch" même si le travail est terminé.

**Preuve :**
- `data/devis.json` contient toutes les soumissions, y compris celles qui ont affiché l'erreur
- Email reçu à 20h16 = confirmation que le pipeline complet s'exécute

**Comportement réel :**
- Le devis est créé ✅
- Le PDF est généré ✅
- Les 2 emails sont envoyés ✅
- La confirmation browser échoue ⚠️ (UX dégradée, pas un bug fonctionnel)

**Cause probable :** Timeout de socket HTTP (~5s sur macOS) pendant la génération Puppeteer.

**Correction à implémenter (prochaine session) :**
Répondre immédiatement `202 Accepted` au navigateur, puis traiter PDF + emails en arrière-plan :
```js
// Dans routes/devis.js
saveDevis(devis);
res.json({ success: true, id: devis.id }); // répondre AVANT le PDF

// Traitement async en arrière-plan (ne bloque plus la réponse)
setImmediate(async () => {
  try {
    const pdfBuffer = await generatePDF(devis);
    await sendDevisEmails(devis, pdfBuffer);
  } catch (err) {
    console.error('Erreur traitement arrière-plan :', err);
  }
});
```
Ce pattern répond en <100ms et élimine le timeout de socket.
