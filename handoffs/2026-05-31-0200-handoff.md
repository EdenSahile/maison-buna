## Blocages / échecs

### Problème central — emails ne partent pas depuis setImmediate du serveur

**Symptôme** : `data/bg-errors.log` montre "BG start" et "PDF OK" mais jamais "Emails OK" ni "ERREUR". La connexion SMTP (puis l'appel API REST) ne complète jamais depuis le processus serveur.

**Cause identifiée** : accumulation de Promises pendantes (ancien code SMTP). Après plusieurs soumissions échouées, le processus serveur accumule des connexions TCP ouvertes vers smtp-relay.brevo.com. Les nouvelles tentatives (SMTP ou fetch HTTP) ne complètent plus.

**Preuve** : à 23:04 UTC (1h après un redémarrage propre, sans Promises pendantes), un email a été envoyé avec succès depuis le serveur. Toutes les tentatives suivantes dans la même instance ont échoué.

**Approches tentées** :
- Nodemailer avec timeouts (`connectionTimeout`, `greetingTimeout`, `socketTimeout`) → pas appliqués (nodemon restart entre-temps)
- Promise.race avec timeout 20s dans routes/devis.js → timeout ne se déclenche pas (serveur chargé avec ancienne version)
- API REST Brevo (fetch) à la place de SMTP → fonctionne parfaitement en test direct, mais hang dans setImmediate du serveur accumulé
- Email envoyé de façon synchrone avant res.json() → crash serveur (exit 52)

**Clé de compréhension** : mes tests directs (`node --input-type=module`) fonctionnent TOUJOURS. Le serveur fonctionne UNIQUEMENT après un redémarrage propre (première soumission). Ensuite les Promises SMTP pendantes bloquent.

**Solution non encore testée** : redémarrer le serveur proprement MAINTENANT avec le nouveau code REST API (mailService.js est déjà réécrit avec fetch). Le code REST API renvoie des erreurs en < 1 seconde (400/401 testés), donc pas de hang possible. L'accumulation de Promises pendantes SMTP serait effacée au redémarrage.
